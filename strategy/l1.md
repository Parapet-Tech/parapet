# L1: Lightweight Lexical Sensor

Naming note: this is the current implementation doc for the compiled SVM,
which maps to target `L2` in `strategy/layers.md`. The code and config still
use legacy `L1` names until the runtime rename lands.

L1 is Parapet's compiled-in first-pass detector for prompt-injection-like text in untrusted messages.

It uses offline-trained character n-gram weights shipped in the binary for very low-latency inference.

## Strategic Role

L1 is a sensor, not the whole defense story.

Its job is to:

- provide very fast first-pass coverage on untrusted message text
- supply a strong lexical signal for routing, observability, and later-layer fusion
- catch obvious prompt-injection language cheaply

Its job is not to perfectly solve every hard benign or context-sensitive case on its own.

## Config

```yaml
layers:
  L1:
    mode: block      # shadow | block
    threshold: 0.0
```

## What L1 Scans

L1 scans message content when `message.trust != trusted`.

1. Trusted messages are skipped.
2. Empty messages are skipped.
3. L1 uses message-level trust, not per-span trust.

## How Scoring Works

1. Raw pass: linear margin from character n-gram features.
2. Squashed pass: second score over alphanumeric-squashed text.
3. Runtime score: the stronger of the raw and squashed views.
4. Calibrated score: sigmoid-mapped probability in `[0.0, 1.0]` for downstream signal use.
5. Verdict: first message with `raw_score >= threshold` is a block candidate in `block` mode.

Even in `shadow` mode, L1 should still score all untrusted messages for observability and downstream routing.

## Modes

1. `block`: block request when verdict is block.
2. `shadow`: log detection but allow request.

## Current Direction

Recent work changed how L1 should be interpreted:

- L1 remains valuable as the fast lexical front door.
- L1 should be optimized for useful routing and coverage, not treated as a standalone final classifier.
- The main remaining benign problems are concentrated in attack-shaped but non-malicious families such as `notinject`, `atlas_neg`, `wildguardmix`, and `bipia`.
- Current corpus work is testing whether benign ratio and benign composition can reduce trigger-word overfire without unacceptable recall loss.

## Holdout Coverage Caveat

The generated public `schema/eval/l1_holdout.yaml` is an in-distribution
holdout for the source files used by the current training script, but it should
not be read as complete language coverage. In particular, the current public
holdout has no Arabic-script rows even though Arabic is part of the active
benign composition target in project-local corpus planning.

L1 metrics reported only against this holdout therefore need an
Arabic-coverage caveat. Arabic behavior should be measured against a reviewable
Arabic eval slice or a regenerated holdout that preserves the intended language
composition before making language-level claims.

Current decision: accept this as a known public-eval limitation and report L1
numbers as measured against the current public holdout. Do not copy local-only
staged, verified, or runner AR rows into the public holdout without a separate
publishability decision.

See `strategy/current_direction.md` for the current program-level summary.
